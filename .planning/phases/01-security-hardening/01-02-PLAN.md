  <action>
    **apps/api-postgres/app/middleware/rate_limit.py — full rewrite (per D-03, SEC-03):**

    1. Replace the entire custom `RateLimitMiddleware` class with a SlowAPI-based implementation.
    2. Import: `from slowapi import Limiter` and `from slowapi.util import get_ipaddr`.
    3. Create a module-level `limiter = Limiter(key_func=get_ipaddr)` instance.
    4. Create a `get_user_key(request: Request) -> str` function that:
       - Tries to extract user from `request.state.user` (set by auth middleware).
       - If user exists, returns `f"user:{user.id}"`.
       - Falls back to `get_ipaddr(request)`.
    5. Create a `RateLimitMiddleware` class (keep the same name for backward compatibility with main.py imports):
       - `__init__(self, app, max_requests: int, window_seconds: int, exclude_paths: list[str] | None = None)`.
       - Store `max_requests` and `window_seconds`.
       - Build rate limit string: `f"{max_requests}/{window_seconds}second"`.
       - In `dispatch`: for excluded paths, pass through. For all other paths, check if `request.state.user` exists — if yes, use user key; if no, use IP key. Apply the rate limit using SlowAPI's internal limiter.
    6. Keep the same JSON response format for 429 errors: `{"error": "RATE_LIMIT_EXCEDIDO", "message": "Muitas requisições. Tente novamente em breve.", "status_code": 429}` with `Retry-After`, `X-RateLimit-Limit`, `X-RateLimit-Remaining` headers.
    7. Keep `structlog` logging for rate limit hits.

    **Alternative simpler approach (preferred):** Instead of wrapping SlowAPI in a middleware class, use SlowAPI's native FastAPI integration:
    - Remove the middleware class entirely.
    - Export `limiter` instance and `_rate_limit_exceeded_handler` from the module.
    - In `main.py`: add `app.state.limiter = limiter` and `app.add_exception_handler(RateLimitExceeded, _rate_limit_exceeded_handler)`.
    - Use `@limiter.limit()` decorators on specific routes. But since we want global rate limiting, the middleware approach is simpler.

    **Recommended approach:** Keep the middleware pattern but replace the internal sliding-window logic with SlowAPI's `limits` library:
    - Use `limits.parse(f"{max_requests}/{window_seconds}second")` to create a rate limit.
    - Use `limits.storage.MemoryStorage()` for in-memory storage.
    - The middleware checks `storage.hit(limit, user_key)` and `storage.get_moving_window(limit, user_key)`.
    - This gives us SlowAPI's battle-tested storage while keeping the middleware pattern that main.py already uses.

    **apps/api-postgres/app/main.py:**
    8. No changes needed if keeping the same middleware class name and constructor signature. The existing `app.add_middleware(RateLimitMiddleware, ...)` call will work.

    **apps/api-postgres/tests/test_rate_limit.py — new file:**

    9. Create integration tests using TestClient:
       - `test_rate_limit_by_ip`: Send `max_requests + 1` requests to a non-authenticated endpoint (e.g., `/health`). Assert the last one returns 429.
       - `test_rate_limit_by_user_id`: Authenticate as user A, send `max_requests + 1` requests to an authenticated endpoint. Assert 429.
       - `test_two_users_independent_limits`: Create two users. User A exhausts their limit. User B from same IP can still make requests. Assert user B gets 200 while user A gets 429.
       - `test_rate_limit_excluded_paths`: Request `/health` repeatedly. Assert never gets 429.
       - `test_rate_limit_headers`: Make a request. Assert response has `X-RateLimit-Limit` and `X-RateLimit-Remaining` headers.
    10. Use small `max_requests` (e.g., 3) in tests for fast execution. Override settings via monkeypatch or dependency injection.
